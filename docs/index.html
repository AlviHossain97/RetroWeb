What it is, in one paragraph

PiStation is two physical hosts cooperating over the LAN so each can do what its hardware is suited for. The Raspberry Pi 3 sits at the TV and runs RetroPie + EmulationStation + RetroArch for native low-latency emulation, Kodi for media playback, and Moonlight as a thin client for the streamed dashboard. The laptop runs the FastAPI analytics backend, MySQL, the React 19 dashboard PWA, the Jellyfin media server, and a local AI stack (Parakeet TDT 1.1B for STT, Kokoro ONNX for TTS, NVIDIA-hosted LLMs for completion). A user sitting on the sofa with one controller can launch a GBA homebrew, watch a movie through Kodi, or open the dashboard — the dashboard is rendered on the laptop's RTX 3070 and streamed to the Pi's TV via Sunshine NVENC, so the experience feels like one device while the heavy lifting happens invisibly on the laptop.

The marker test: after reading this page and looking at the GitHub directory tree, a marker should be able to say "Pi-and-laptop system, three interaction paths, original homebrew game corpus, and a streamed dashboard with local AI." If they can say that without reading the report, this site is doing its job.

Why split the work?

The Pi 3 is excellent at low-latency native emulation and HEVC/H.264 video decode — the workloads it was designed for. It's poor at GPU-bound modern web rendering, AI inference, and real-time video transcoding. The laptop is the inverse — strong at GPU work, weak at being plugged into a TV. Sunshine + Moonlight bridges them so the user gets a unified single-device experience while the laptop's GPU transparently does the heavy lifting. This matches the literature reviewed in the report (Rzepka, Gamess, Suder) which warns against forcing modern frontend workloads onto Pi-class hardware.

The three interaction loops

One physical interface (TV + controller), three completely different workloads — all unified by EmulationStation as the launcher.

1. Native gaming Operational

EmulationStation lists every system on the Pi (GBA, SNES, NES, etc.). The user picks a game; RetroArch launches it natively on the Pi using the appropriate libretro core (lr-mgba for the three GBA homebrew titles in this submission). RetroPie's runcommand-onstart.sh and runcommand-onend.sh hooks fire before and after each game session and POST session metadata (ROM path, system, emulator, core, hostname, started/ended timestamps, duration) to the FastAPI backend on the laptop, which writes to MySQL.

2. Media (Jellyfin → Kodi) Operational

Jellyfin is registered as an EmulationStation system entry, appearing in the carousel alongside SNES and GBA. Selecting it runs ~/RetroPie/roms/jellyfin/Jellyfin.sh, which delegates to /usr/bin/kodi-standalone (the distro's official Kodi-as-a-session wrapper). Inside Kodi, the Jellyfin for Kodi add-on connects to the Jellyfin server on the laptop. Library scanning and any transcoding happen on the laptop's GPU; the Pi handles HEVC/H.264 hardware decode for playback.

3. Dashboard (Sunshine ↔ Moonlight) Operational

The React 19 + Vite 7 + Tailwind v4 PWA runs in a fullscreen Chromium window on the laptop. Sunshine NVENC captures that window and streams it as H.264/HEVC to moonlight-qt on the Pi at 720p / 30 fps / 8 Mbps / software-decoded. The Pi 3 cannot render a modern React PWA, but it can decode hardware video — so the dashboard "appears" on the TV without the Pi doing the heavy work. Controller and keyboard input flow back over the same Sunshine connection, surfaced to the React app as standard Gamepad-API events.

End-to-end session walkthrough

A concrete walkthrough of all three loops touching one piece of state — what actually happens between picking a game and seeing it as a tile on the dashboard.

  1. step 1User selects a game in EmulationStationFrom the sofa with the 8BitDo controller, picks BastionTD.gba from the GBA carousel.
  2. step 2RetroPie's runcommand firesruncommand-onstart.sh shells out to ~/pistation/session_logger.py, which POSTs {pi_hostname, rom_path, system_name="gba", emulator="retroarch", core="mgba", started_at} to http://<laptop-host>:8000/session/start. FastAPI inserts a row into the sessions table; logger stashes the assigned session_id in /tmp/pistation_session.json.
  3. step 3RetroArch launches lr-mgba natively on the PiROM read from local SD or (when CIFS mount is wired) from the laptop's Samba share. Game runs at full speed on the Pi's hardware — no emulation overhead from the laptop side.
  4. step 4User plays for 12 minutes; presses select+start to exit
  5. step 5runcommand-onend firessession_logger.py reads the state file and POSTs {session_id, ended_at, duration_seconds=720} to http://<laptop-host>:8000/session/end. FastAPI updates the row.
  6. step 6User opens Dashboard from the RetroPie system menuRetroWeb.sh resolves the laptop's hostname, then xinit + matchbox-window-manager + moonlight-qt. Sunshine on the laptop accepts the connection.
  7. step 7The "Recently played" tile shows BastionTD with a 12-min entryReact reads from the same MySQL row the runcommand hook wrote, via FastAPI's /stats/recent.
  8. step 8User pushes-to-talk: "How long did I play Bastion this week?"Parakeet transcribes locally → AI context service queries the sessions table → NVIDIA LLM produces an answer streamed back to Kokoro for TTS → user hears the reply on the TV speakers.

That flow is the artefact. Each component below documents the implementation behind its part of the path.

Voice pipeline

End-to-end flow for a single spoken turn. Audio leaves the browser as PCM16 16 kHz mono and returns as PCM16 22.05 kHz mono — both in 80 ms frames for smooth playback.

  Voice Gateway WebSocket  (FastAPI :8000)
       │   input_audio.chunk · input_audio.stop (silence cutoff)
       ▼
  Parakeet STT  (NeMo TDT 1.1B, fp16 on CUDA, :8786)
       │   { text: "..." }
       ▼

       │

       │   delta tokens
       ▼

       ▼

       ▼
       Speaker

Invariants

  • VAD suppression while TTS plays — the frontend gates the mic pipeline on TTS playback state, so the assistant can't transcribe its own voice in auto_near_field mode.
  • Provider fallback — VOICE_PROVIDER_ORDER = legacy,voicechat,hosted. If the primary provider's health() fails at session start, the gateway falls back and emits a provider.changed event.
  • Session timeout — VOICE_SESSION_MAX_SECONDS default 840 s. A session.expiring event fires 5–30 s before the cutoff so the client can reconnect cleanly.
  • Turn cancellation — frontend can send response.cancel; the gateway sets a cancel event that interrupts both LLM streaming and TTS chunk delivery.
  • Text-chat TTS — typed messages also get voiced. createTTSSession() in useChatVoice.ts flushes at sentence boundaries so playback starts while the LLM is still generating.

AI & models

Multi-model LLM picker

All seven LLMs go through NVIDIA's integrate.api.nvidia.com/v1/chat/completions endpoint with streaming enabled. Models are configurable in src/routes/chat/constants.ts; adding a new one is "append to NVIDIA_MODELS + drop an icon at public/model-icons/".

  • DeepSeek V4 Pro Default — reasoning-heavy replies
  • Kimi K2 Thinking — long-context reasoning
  • Gemma 3 27B · Mistral Large 3 675B · Step 3.5 Flash · MiniMax M2.7 · GLM 4.7

Local AI services

STT
NVIDIA NeMo Parakeet TDT 1.1B, fp16 on CUDA, ~8 GB VRAM (fallback to parakeet-tdt-0.6b-v2 via PARAKEET_MODEL env).
TTS
kokoro-onnx v1.0 with voices-v1.0.bin (354 MB cache). ONNX runtime, optional CUDA EP.
Web grounding
Optional Tavily or self-hosted SearXNG. WEB_SEARCH_MODE = auto/always/never.
Image generation
Optional ImageRouter API (google/nano-banana-2:free default).
Why local STT/TTS? The user's voice never leaves the laptop. Only the transcribed text reaches NVIDIA's API. This was a deliberate privacy choice — sending audio to third-party services would have been simpler but contradicts the "self-hosted by default" property documented in the project's design goals.

Why software decode on the Pi 3?

The Pi 3's hardware H.264 path can be unstable on RetroPie 4.8's older kernel. Software decode at 720p30 with 8 Mbps stays within the Pi 3's CPU envelope and produces a stable frame rate; the trade-off is bitrate ceiling (raising past ~10 Mbps drops frames). The launcher script RetroWeb.sh in ~/RetroPie/retropiemenu/ sets this explicitly via moonlight-qt --720 --fps 30 --bitrate 8000 --video-decoder software --video-codec H.264.

End-to-end latency

NVENC encode ≈ 5 ms/frame, LAN transit on wired Ethernet sub-millisecond, software decode on Pi 3 ≈ one frame at 30 Hz. Pi-perceived end-to-end latency stays below ~50 ms — well within the threshold where controller input feels responsive on the TV.

Original GBA homebrew corpus

Three Game Boy Advance homebrew titles, written from scratch for this project so the test corpus is IP-safe — no commercial ROMs are distributed in this repository, and the games are themselves part of the artefact rather than third-party content used to demonstrate it.

The Python-to-GBA-native pipeline

Every game in this corpus followed the same pipeline: rapid Python prototype on the laptop to iterate gameplay design (lane logic, fuel/repair pickups, enemy waves, tower placement) without memory or render budgets, then a port to GBA-native C or C++ that runs on the Pi via lr-mgba under hard hardware constraints — no FPU (fixed-point math), 240×160 framebuffer, no dynamic allocation in hot paths, sprite tile/palette layout, ROM mapping. The two implementations share no source. What carries across is the design.

Audio infrastructure is preserved in the source where it exists (Bastion TD's SfxId/BgmId enums, SDL2Audio/GbaAudio implementations, app.audio→play_sfx() call sites; Mythical and Bastion TD's Python audio_manager.py) as documented engineering work, but no audio playback occurs at runtime in the shipped artefact. What didn't ship — and why

Every project has these. Captured here so the marker doesn't have to infer:

  • Library page — route exists in src/routes/, marked "coming soon". Cut from scope to keep voice and analytics polish in.
  • Cloud save sync — save files stay in the SRAM of each .gba ROM and on the Pi's SD card.
  • Per-user profiles — single user only; multi-user scoped out at the December mid-point.
  • Netplay matchmaking — long-term roadmap.
  • Audio in shipped GBA builds — corpus-wide silent-audio policy. Audio infrastructure is preserved in source as documented engineering work but disabled at build time, eliminating third-party audio attribution complexity.

These are explicit deferrals, not unfinished features. Each is documented in the report's evaluation section with the reason it was cut.
